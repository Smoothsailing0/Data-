# 公共管理 C刊写作

## Overview

本 skill 面向中文公共管理、公共行政与治理研究的 C刊/CSSCI 论文写作。它不是普通润色器，也不是终稿自检清单，而是把 C刊论文中最容易失控的“问题意识、文献推进、理论机制、方法边界、结果解释和贡献收束”转化为可执行的写作规则。

适用任务包括题目打磨、摘要写作、引言重写、文献综述、理论框架、机制解释、研究设计、结果分析、讨论结论和反模板化修改。若用户要求投稿前整体审查，应配合或改用 `c-journal-paper-self-review-zh`。

## Core Principle

C刊写作不是把句子写得更像论文，而是让每一段承担清楚的论证功能。正式写作前必须先判断当前段落在全文中的作用：它是在提出问题、承接文献、界定概念、进入案例、说明方法、解释结果，还是收束贡献。只有先确认功能，才能决定开头方式、材料密度和评述位置。

不要把“判断先行”误解为每段都用同一种显性判断句开头。相邻段落应形成“总起判断—承接展开—评价转向—贡献收束”的推进关系，而不是连续出现“关键在于”“核心在于”“问题在于”的模板化起笔。

## Mandatory Workflow

生成或改写正文前，先在内部完成以下判断：

1. 文类判断：摘要、引言、文献综述、理论框架、研究设计、结果分析、讨论或结论。
2. 问题判断：本段要回答的研究问题是什么，是否只是背景介绍。
3. 段落功能：本段承担起势、承接、转向、证明、比较、深化还是收束。
4. 材料落点：本段至少应落到政策节点、文献发现、案例过程、访谈材料、变量题项、统计结果或机制环节中的一类。
5. 评述位置：材料之后必须说明这些材料改变了什么判断，而不是只写“具有重要意义”。
6. 反模板检查：输出前删除流程式开头、作者排队、对偶句、空泛拔高和政策口号。

## Argument Unit Map

写一个小节、二级标题或连续多段正文前，先形成论证单元地图。地图不必每次展示给用户，但正文必须服从它。

论证单元地图至少包括四项：

1. 本单元处理的核心问题。
2. 准备分几段推进。
3. 每段分别承担问题提出、材料展开、文献转向、机制解释、结果解释或贡献收束中的哪一种功能。
4. 段落之间依靠什么关系推进，如问题链、政策过程、经验场景、方法边界、统计结果或机制链条。

若相邻三段都采用相同句式起笔，即使意思正确，也默认需要改写为不同功能的段落入口。

## Opening Modes

根据段落功能选择开头方式，不使用万能背景句。

1. 问题压缩式：用于引言、研究困惑和机制解释。首句压出具体治理难题或解释缺口。
2. 文献转向式：用于文献综述。首句写已有研究焦点如何移动，而不是作者姓名排队。
3. 概念限定式：用于核心概念和理论框架。首句划清概念边界或理论分工。
4. 方法边界式：用于问卷、案例、访谈、回归、fsQCA、指标体系和文本分析。首句说明方法在本文中解决什么问题，不能替代什么判断。
5. 经验场景式：用于案例和基层实践。首句从主体、事务、空间、组织或服务场景进入。
6. 结果解释式：用于表格、模型、组态路径和评价结果。首句先说明结果回答哪个问题，再报告关键发现。
7. 贡献收束式：用于讨论和结论。首句提炼本文推进了什么理解，而不是重复前文。

禁止默认用“随着……不断推进”“在……背景下”“近年来”“本文首先……其次……”开头。

## C Journal Abstract Rules

C刊摘要不是引言缩写，也不是“背景、方法、结论、意义”的机械拼接。摘要必须作为一篇论文的压缩论证单元，交代研究问题、研究对象、材料或方法、核心发现和学术贡献。

摘要写作前先判断全文最小贡献是什么。摘要至少回答五个问题：

1. 研究问题：本文处理哪个治理难题、解释缺口或经验矛盾。
2. 研究对象：研究发生在哪类制度场景、区域、主体关系或政策过程之中。
3. 方法或材料：案例、问卷、访谈、政策文本、统计模型、fsQCA 或综合评价在本文中承担什么功能。
4. 核心发现：至少给出两项可辨识的发现或机制判断，不能只写“具有显著影响”。
5. 贡献含义：说明本文对概念边界、机制解释、政策理解或中国情境研究推进了什么。

若用户没有给出期刊要求，中文 C刊摘要通常控制在 300-500 字。摘要首句优先从研究困惑或具体对象进入，不用宏大背景句。

推荐顺序：研究困惑或治理难题—研究对象与材料—方法功能—核心发现或机制判断—理论/政策含义。

## Introduction Rules

引言的任务不是证明议题重要，而是把重要议题压成可研究的问题。高质量引言应完成四步：

1. 从具体治理现象、政策变化或经验矛盾进入。
2. 说明既有治理解释或研究视角为什么不足。
3. 推出本文的研究问题、对象和分析入口。
4. 交代本文可能形成的理论、方法或经验贡献。

政策文件可以说明制度环境，但不能替代问题意识。不要用一长串政策背景代替“为什么需要研究这个问题”。

## Literature Review Rules

文献综述按研究推进组织，不按作者姓名排队。段落主语优先是“研究焦点、解释线索、争论、方法转向、场景差异或问题推进”，不是作者。

写作要求：

1. 每段先概括一类研究处理了什么问题，再嵌入代表性文献。
2. 每段至少落到两类具体内容：研究对象、案例区域、样本数据、方法工具、关键发现、机制解释或比较结论。
3. 作者名只能作为证据节点出现，不能连续使用“某某指出、某某认为、某某发现”组织段落。
4. 每个小点结尾必须说明已有研究推进到哪里，本文为何还要继续处理。
5. 不提前展开本文的变量拆分、模型设定、案例选择和指标操作化。

常用段落类型：研究推进型、机制解释型、方法转向型、场景拓展型、述评收束型。

## Theory And Mechanism Rules

理论框架不能写成概念介绍。每个理论都要说明它解释本文哪一部分，也要说明它不负责解释什么。若使用多个理论，必须写出分工，而不是并列摆放理论名称。

机制解释必须写清主体、规则、资源、行动过程、反馈和结果之间的关系。不要用“技术赋能、组织协同、资源整合”替代机制本身。

机制段落优先回答：

1. 谁在行动。
2. 在哪个制度或服务环节行动。
3. 依靠什么规则、资源或技术条件行动。
4. 行动如何改变服务、治理或参与结果。
5. 机制失败时会出现什么边界或后果。

## Research Design Rules

方法段服务研究问题，不服务方法本身。写问卷、访谈、案例、组态分析、综合评价、结构方程、回归模型或文本分析时，先交代分析单位、样本来源、变量、题项或材料为什么能回答研究问题，再交代操作步骤。

不要把方法段写成统计教材。公式、阈值和检验结果可以出现，但必须说明它们在本文中承担什么功能：筛题、赋权、比较、识别路径、解释差异，还是验证一致性。

## Result Interpretation Rules

结果分析不能只复述表格。先说明该表、模型或路径要回答的问题，再报告关键数值，最后解释这些数值改变了我们对研究对象的什么认识。

若结果与预设不同，不急着解释为数据异常。公共管理研究中的偏离常来自制度执行、主体能力、地方情境或测量口径差异，应判断这种偏离是否揭示了新的治理边界。

## Discussion And Conclusion Rules

讨论不是重复结果，贡献不是自我表扬。讨论应把经验发现推回理论、方法或政策问题，说明本文发现如何修正、补充或细化已有解释。

结论应先提炼研究判断，再说明理论意义、方法意义和政策含义。政策建议必须从结果中长出来，不能另起炉灶写成通用对策。

结尾段避免“综上所述”式复述。更好的结尾是说明本文把某个问题推进到了哪里，以及这种推进的边界是什么。

## Anti-Template Rules

默认重写以下表达：

1. “随着……不断推进”“在……背景下”“近年来”作为开头。
2. “不是A，而是B”“并非A，而是B”“不在于A，而在于B”高频出现。
3. “一方面……另一方面……此外……”作为段落骨架。
4. “具体来看”“总体来看”“由此可见”“值得注意的是”连续使用。
5. “具有重要意义”“提供理论支撑”“奠定基础”等没有具体对象的评价。
6. “赋能、驱动、融合、重塑、抓手、闭环、图景、路径”等抽象词连续堆叠。
7. 用冒号、引号或破折号制造总领感，而不是用正常句法解释关系。
8. 文献综述中连续“某某指出、某某认为、某某发现”。

替代原则：

1. 从具体治理难题、方法边界、制度过程、数据结果或案例环节起笔。
2. 用主体、资源、规则、流程、反馈和结果替代抽象口号。
3. 让转折来自材料本身，不靠路标词硬转。
4. 保留必要证据，不为追求工整删掉有效的机制细节。

## Output Contract

生成正式论文正文时，默认输出两部分：

1. 正文。
2. 写作逻辑与风格说明，控制在 3-6 行，说明本次选择的起笔方式、材料推进方式和去模板处理。

如果用户明确要求“只给正文”或“不要说明”，则省略第二部分。

若本轮只提供结构方案、段落诊断、标题判断、修改建议或优劣比较，也必须附上简短判断依据，说明依据的是章节功能、材料密度、文献推进、测量逻辑、机制解释或反模板规则中的哪几项。

## Source And Citation Rules

涉及具体文献、政策、数据或案例时，优先使用用户提供材料、正式政策文件、学术数据库和已核实来源。政策文件用于说明制度环境，学术文献用于支撑概念、机制、方法和研究缺口。

若用户要求“先读文献再写”，先列出本轮参照来源，再写正文。若用户要求插入引用，按其指定格式处理。

## Final Check

交付前逐项检查：

1. 是否先判断段落功能，再写正文。
2. 是否把公共管理对象写清楚，而不是停留在宏大背景。
3. 材料是否形成问题链、过程链、因果链或机制链。
4. 是否在材料节点后加入作者评述。
5. 是否删除流程式开头、镜像条件句、对偶式并列和空泛拔高。
6. 文献综述是否按研究推进组织，而不是按作者姓名组织。
7. 方法段是否说明方法功能，而不是复述教材。
8. 结果解释是否回答研究问题，而不是复述表格。
9. 结论是否提炼了贡献边界，而不是重复前文。
10. 正式正文后是否提供具体写作逻辑与风格说明，除非用户要求只给正文。
